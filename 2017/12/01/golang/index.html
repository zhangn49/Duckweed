Golang, 人生需要够浪

Dec 1, 2017

机缘巧合

很早之前就听过了Golang的大名，第一感觉由Google开发的语言应该怎么都不会太差，但一直都没机会使用。由于收到了春季ad，在大学毕业前夕投递了两家公司，最后到了b站做实习生。本来我所在的组很喜欢用openresty，但是我在实际使用过程中发现http的settimeout有一些问题，因此转而使用go。

初识够浪

Golang的官网做的相当不错，有一个A Tour of Go的教程，介绍了一些go的语法和特性，上手起来很快。在这里我要简单说一下从cpp到go的一些体会。

  1. 环境配置相当方便，只需要下载二进制解压，添加GOPATH即可。开发用的文件夹层次分明，自带包管理工具。如果使用glide来管理依赖的包，可以做到将所依赖的包放在目录的vendor文件夹下，别人在GitHub上下载你的项目后可以即时编译。
  2. 部署项目非常方便，编译二进制可以选择平台和操作系统，编译后的二进制可以直接运行，因此go的一大应用场景就是后台开进程提供服务。
  3. 由于不使用分号作为语句的结束，强行使得大括号不换行，举双手赞同。同时else也强制不换行了。另外可以go format来格式化go的代码文件，但是会把import的package名按字典序排列，我有把package名从短到长排列的强迫症，所以基本不会用。
  4. 变量类型放在变量名后，这个点被王垠大神喷了，说是用眼球来parse类型，但我感觉很不错，有时候需要思考这个变量是什么类型，先写出来变量名反而易于思考，不知道是不是我的脑回路过于清奇了。
  5. 变量声明可以用关键字var，也可以使用 := 省略var。golang作为一个类Java语言，代码中主要使用驼峰命名法，并且结构体和package中的变量大写开头为public，小写开头为private。
  6. 空值用nil而非null，数据结构基本靠Slice和Map搞定。Slice可以看作是不定长度的数组，底层实现非常简单，是一个带有len，cap，以及指向array的ptr。

自带并发

Golang的最大的优点可以说是自带并发，由goroutine协程来实现。使用时用go关键字来触发，比起传统的多线程首先在coding上来说免去了回调这个操作。并发的同步一般情况下依靠channel来实现，channel可以看作是线程安全的一个容器，作为管道，用左箭头<-来支持单向的push和pop。在声明channel时可以指定它的大小，当channel为空的时候可以阻塞当前协程。在sync中，还有一些别的关于并发的数据类型，例如WaitGroup等。可以用runtime包的函数GOMAXPROCS来指定运行时最大的cpu数。

谈到并发，就不得不提线程安全。channel是一个线程安全的数据结构，其内部实现主要是靠mutex，需要看channel的源码来理解，这里要先挖个坑了。其次很重要的一点是gorountine的调度问题，算作是第二个坑。立个Flag这个月内要把挖好的两个坑填起来。

遇到的坑

虽然我非常推崇Golang，喜欢它的一些语言特性和简单灵活的部署，但是在使用过程中遇到的坑还是非常多的，在这里做个小小的总结。

  1. 自带的map并不是线程安全的。在Golang1.9之前，线程安全的map需要通过map加一把读写锁来实现，值得一提的是，我由于担心锁的性能消耗，曾经尝试了对每一个kv加一把锁，当lock其中一个key时，对别的kv不影响。但是现实是残酷的，或许小并发下读写没什么问题，但是并发一起来就会造成iowait报错，程序crash。其实锁操作本身的性能损失是在微秒级别的，对一个map上一把读写锁性能在99%的场景下够用了。Golang1.9官方在sync包提供了Map，支持并发读写操作。
  2. http读取json接口占用内存过大。原因在于http的body读取使用了body, err := ioutil.ReadAll(resp.Body), ReadAll初始申请521bytes, 申请内存不足时会以2倍速度增长, 而golang申请的内存使用完毕后不会及时归还给操作系统。
    申请bytes.NewBuffer, 然后使用io.Copy来读取。另外一个点就是再已知map大小的情况下，make map时指定capacity。
  3. http请求需要复用transport。根据golang官方文档: Clients and Transports are safe for concurrent use by multiple goroutines and for efficiency should only be created once and re-used. 当用协程开启多个http请求时，需要复用Transport，可以减少内存和cpu的使用。
  4. Golang中所有的参数传递均为按值传递。由于slice和map本身储存的信息是指针，因此传参时无需传引用。使用过程中还算友好，需要注意的有sync包中的WaitGroup的传递需要传引用，不然不会生效。
  5. 垃圾回收会引起stw，即stop the world。stw主要会出现在扫描对象这个阶段，由于垃圾回收基于引用计数，因此struct中存int类型不会被扫描到，但是存int*指针的话会被扫描到。因此需要对程序的应用场景评估之后，选择struct成员的存储方式。

小结

通过实习期间对Golang的接触，最大的感觉就是码起来特别爽，有一种行云流水的感觉。而且因为开发一直用远程的测试虚拟机，因此一直在terminal的ssh下进行编程，对linux的熟悉程度上升了一个台阶，对vim也从畏惧变成了喜爱。关于Golang还有几个topic我想讨论，主要是interface，gorountine的调度，反射，以及垃圾回收。坑已经挖好了，慢慢填。
